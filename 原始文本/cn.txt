一、计算机网络基础
1.1 网络分层模型
1.1.1 OSI 七层模型（国际标准化组织制定）
物理层：透明地传送比特流，负责传输 0/1 二进制信号，定义接口、电压、速率等物理规范。
数据链路层：将比特流封装成帧，负责帧编码、差错纠正、物理寻址（MAC 地址），控制相邻节点间的数据传输。
网络层：负责路由与寻址，决定数据在网络中的传输路径，实现跨网络的数据包转发。
传输层：为两台主机进程之间的通信提供通用数据传输服务，负责端到端的连接、可靠性控制。
会话层：管理应用程序之间会话的建立、维护、断开与重连，控制通信的交互节奏。
表示层：负责数据处理，包括编解码、加密解密、压缩解压缩，保证数据能被正确识别。
应用层：直接为计算机用户提供网络服务，是用户与网络的接口，承载各类应用协议。
1.1.2 TCP/IP 四层模型（实际互联网主流模型）
应用层：对应 OSI 的应用层 + 表示层 + 会话层，实现具体的应用功能。
传输层：与 OSI 传输层一致，提供端到端传输服务。
网络层：与 OSI 网络层一致，负责路由寻址。
网络接口层：对应 OSI 的数据链路层 + 物理层，负责物理传输与帧封装。
1.1.3 网络分层的核心原因
各层相互独立：每层只需关注自身功能，无需关心下层实现，通过接口调用下层服务。
灵活性与可替换性强：每层可独立升级、替换技术，不影响整体网络结构。
复杂问题拆解：将庞大的网络问题拆分为小而清晰的子问题，降低设计、实现与标准化难度。
1.2 各层常见协议
应用层：HTTP、HTTPS、FTP、SMTP、POP3、IMAP、Telnet、SSH、DNS、RTP、RTCP、DHCP、SNMP、NTP
传输层：TCP、UDP、SCTP
网络层：IPv4、IPv6、ARP、ICMP、IGMP、NAT、OSPF、RIP、BGP
网络接口层：以太网、Wi-Fi、PPP、MAC 协议
二、传输层核心：TCP 与 UDP
2.1 TCP 与 UDP 详细区别
连接性
TCP：面向连接，传输前必须通过三次握手建立连接，传输后通过四次挥手释放连接。
UDP：无连接，发送数据前无需建立连接，直接发送数据报。
可靠性
TCP：可靠传输，通过序列号、确认应答 ACK、超时重传、流量控制、拥塞控制，保证数据无差错、不丢失、不重复、按序到达。
UDP：不可靠传输，仅尽力交付，不保证数据到达、顺序，无自动重传与确认机制。
状态维护
TCP：有状态，两端需维护连接状态（序列号、窗口大小、已发送 / 已确认数据等）。
UDP：无状态，不维护任何连接信息，开销极小。
传输效率
TCP：开销大、效率低，需建立连接、确认、重传等操作。
UDP：结构简单、开销小、传输速度快。
传输形式
TCP：面向字节流，将数据视为无结构字节流，可拆分 / 合并数据。
UDP：面向报文，保留应用层数据边界，不拆分、不合并数据块。
首部开销
TCP：头部最小20 字节，带选项最大60 字节。
UDP：头部固定8 字节，极简。
通信模式
TCP：仅支持点对点单播。
UDP：支持单播、一对多多播、一对所有广播。
应用场景
TCP：要求数据绝对完整、准确的场景。
UDP：要求低延迟、高实时性，可容忍少量丢包的场景。
2.2 协议选择原则
选择 TCP 的场景（数据完整性优先）
Web 浏览（HTTP/HTTPS）：网页、图片、脚本需完整加载。
文件传输（FTP、SCP）：文件字节不允许丢失 / 错序。
邮件收发（SMTP、POP3、IMAP）：邮件内容必须完整无误。
远程登录（SSH、Telnet）：命令与响应需精准传输。
选择 UDP 的场景（实时性优先）
实时音视频（VoIP、视频会议、直播）：少量丢包仅导致短暂卡顿，优于重传延迟。
在线游戏：位置、状态需极速传输，旧数据无价值。
DHCP：客户端无 IP，无法建立 TCP 连接，且需广播。
IoT 传感器上报：少量数据丢失不影响整体趋势。
2.3 基于 TCP/UDP 的应用层协议
基于 TCP（可靠传输）
HTTP/1.x、HTTP/2、HTTPS：网页传输
FTP、SFTP：文件传输
SMTP、POP3、IMAP：邮件收发
Telnet、SSH：远程登录
基于 UDP（高速传输）
HTTP/3（基于 QUIC）：新一代网页传输
DNS：域名解析（默认 UDP，大包 / 区域传送用 TCP）
DHCP：动态分配 IP
RTP/RTCP：实时音视频
TFTP：简易文件传输
SNMP：网络管理
NTP：时间同步
2.4 TCP 核心机制
2.4.1 三次握手（建立连接）
目的：确认客户端与服务器的发送、接收能力均正常，同步初始序列号。
常见面试题：为什么必须三次？两次无法确认客户端接收能力；四次冗余。
2.4.2 四次挥手（断开连接）
目的：保证双方数据传输完毕，安全释放连接。
常见面试题：为什么不能三次？服务器需先回复 ACK，待数据发完再发 FIN，无法合并。
2.4.3 TCP 可靠性保障机制
序列号与确认应答（ACK）：保证数据按序、无重复。
超时重传：数据包丢失则超时后重发。
流量控制：接收方通过窗口大小告知发送方发送速率，避免溢出。
拥塞控制：网络拥堵时降低发送速率，避免丢包加剧。
三、网络层核心协议
3.1 IP 协议
3.1.1 核心作用
定义数据包格式，负责路由与寻址，让数据包跨网络到达目标设备，是互联网通信的基础。
3.1.2 IPv4 与 IPv6 详细对比
地址长度
IPv4：32 位，约 42 亿个地址，已耗尽。
IPv6：128 位，地址空间近乎无限。
地址格式
IPv4：点分十进制，如 192.168.1.1。
IPv6：冒分十六进制，可省略前导零，如 2001:db8::1。
IPv6 核心优势
无状态地址自动配置（SLAAC）：无需 DHCP 即可自动获取地址。
无需 NAT：地址充足，每个设备可分配公网 IP。
报头简化：处理开销更低，性能更高。
扩展头：支持更多功能。
ICMPv6：优化邻居发现、路径 MTU 发现。
3.2 网络层关键技术
3.2.1 IP 地址过滤
定义：限制 / 阻止特定 IP 或 IP 段访问，是基础网络安全手段。
局限：单独使用无法保证安全，需结合认证、加密。
3.2.2 获取客户端真实 IP
应用层：X-Forwarded-For 请求头，简单但易伪造，仅适用于 HTTP/SMTP。
传输层：TCP Options 字段、Proxy Protocol，适用于所有 TCP 协议，需双方改造。
网络层：隧道 + DSR 模式，适配所有协议，但实施复杂。
3.2.3 NAT（网络地址转换）
作用：将内网私有 IP映射为公网公有 IP，实现多设备共享一个公网 IP 上网。
价值：缓解 IPv4 地址短缺，隐藏内网拓扑，提升内网安全。
3.3 MAC 地址与 ARP 协议
3.3.1 MAC 地址
全称：媒体访问控制地址，**48 位（6 字节）** 物理地址，设备唯一标识。
特性：由 IEEE 统一分配前 24 位，厂商分配后 24 位，全球唯一，永久绑定网卡。
特殊地址：FF-FF-FF-FF-FF-FF（广播地址）。
3.3.2 ARP 地址解析协议
核心作用：解决网络层 IP 地址 → 链路层 MAC 地址的转换问题，保证数据包在物理链路上正确传输。
3.4 ICMP 协议与 PING 命令
3.4.1 ICMP 协议
作用：传输网络状态、错误消息，用于网络诊断与故障排查。
3.4.2 PING 命令
原理：基于 ICMP，发送Echo Request（类型 8），目标回复Echo Reply（类型 0）。
功能：测试主机连通性、测量网络延迟（RTT）、统计丢包率。
3.5 常见路由协议
OSPF：内部网关协议，链路状态算法，按带宽 / 延迟选最优路径。
RIP：内部网关协议，距离向量算法，按跳数选路径。
BGP：边界网关协议，用于跨自治系统的路由交换。
四、应用层核心协议
4.1 HTTP 全系列协议详解
4.1.1 HTTP 各版本演进
（1）HTTP/1.0
短连接：每次请求新建 TCP 连接，效率低。
缓存机制简陋：仅支持 If-Modified-Since、Expires。
无 Host 头：不支持虚拟主机。
不支持断点续传，浪费带宽。
（2）HTTP/1.1（主流）
长连接（Keep-Alive）：默认开启，复用 TCP 连接。
丰富缓存策略：新增 ETag、If-Match、If-None-Match。
支持断点续传：Range 请求头，返回 206 状态码。
新增 Host 头：支持单 IP 多域名虚拟主机。
新增 24 种状态码，完善错误处理。
（3）HTTP/2
多路复用：单 TCP 连接并发传输多请求，解决应用层队头阻塞。
二进制帧：替代文本格式，传输更高效。
头部压缩（HPACK）：减少头部开销。
服务器推送：主动推送客户端所需资源。
缺陷：仍受TCP 层队头阻塞影响，单个丢包阻塞全连接。
（4）HTTP/3（革命性升级）
底层协议：基于UDP+QUIC，弃用 TCP。
解决队头阻塞：流间独立，单流丢包不影响其他流。
连接建立：0-RTT/1-RTT，远快于 HTTP/2 的 2.5-3.5 RTT。
连接迁移：网络切换（Wi-Fi→移动网）不断连。
内置 TLS1.3：全程加密，安全性更高。
头部压缩：QPACK 算法，比 HPACK 更高效。
4.1.2 HTTP 与 HTTPS 区别
端口：HTTP 默认 80，HTTPS 默认 443。
安全性：HTTP 明文传输，易窃听 / 篡改；HTTPS 基于 SSL/TLS 加密，防窃听、篡改、冒充。
资源消耗：HTTPS 加密解密消耗更多服务器资源。
SEO：搜索引擎优先收录 HTTPS 站点。
4.1.3 HTTP 队头阻塞详解
HTTP/1.1：应用层阻塞，单连接串行请求，前一个请求未完成，后续全部阻塞。
HTTP/2：传输层阻塞，TCP 要求按序交付，单个丢包阻塞整个连接的所有流。
HTTP/3：QUIC 彻底解决流间阻塞，仅阻塞丢包的单个流。
4.1.4 HTTP 状态码分类
1XX：信息性，请求正在处理。
2XX：成功，请求正常处理（200 成功、206 部分内容）。
3XX：重定向，需额外操作（301 永久重定向、304 未修改）。
4XX：客户端错误（400 参数错误、401 未授权、403 禁止、404 未找到）。
5XX：服务器错误（500 服务器异常、502 网关错误）。
4.1.5 HTTP 常见请求头
Host：服务器域名与端口。
Connection：连接方式（keep-alive/close）。
Content-Type：请求体格式（application/json、form-data 等）。
Cache-Control：缓存控制（no-cache、max-age）。
Cookie：携带客户端 Cookie。
User-Agent：客户端标识。
4.2 HTTP 状态保持机制
HTTP 本身无状态，无法识别连续请求是否来自同一用户，需通过以下方案保持状态：
4.2.1 Cookie + Session（主流）
客户端登录，服务器验证通过后创建 Session，生成唯一 SessionID。
服务器通过 Set-Cookie 将 SessionID 返回客户端。
客户端后续请求自动携带 Cookie，服务器通过 SessionID 匹配用户状态。
Session 存储：内存（简单）、数据库（持久）、Redis（分布式主流）。
安全配置：Cookie 设 HttpOnly、Secure，防止窃取。
4.2.2 URL 重写（Cookie 禁用时备用）
将 SessionID 拼接在 URL 后，如http://xxx?sessionid=xxx。
缺陷：URL 暴露、安全性低、不友好 SEO。
4.2.3 Token/JWT（无状态认证）
适用于前后端分离、微服务。
流程：登录成功→服务器签发 JWT→客户端存储→请求头携带→服务器验证。
优势：无状态、分布式友好、无需服务器存储 Session。
4.3 基础概念区分
4.3.1 URI 与 URL
URI：统一资源标识符，唯一标识一个资源。
URL：统一资源定位符，是 URI 的子集，既标识又定位资源（指明访问路径）。
4.3.2 GET 与 POST
语义：GET 查询资源，POST 创建 / 修改资源。
幂等性：GET 幂等（多次请求无副作用），POST 非幂等。
参数位置：GET 参数在 URL，POST 在请求体。
缓存：GET 可缓存，POST 不缓存。
安全性：HTTP 下均明文，HTTPS 下均加密；GET 参数更易泄露。
4.4 实时通信协议
4.4.1 WebSocket
基于 TCP，全双工长连接，一次握手后永久通信。
协议前缀：ws://、wss://（加密）。
应用：弹幕、实时聊天、在线游戏、协同编辑。
优势：低延迟、轻量头部、双向主动通信。
4.4.2 SSE（服务器推送事件）
基于 HTTP，单向通信，仅服务器→客户端推送。
优势：实现简单、原生自动重连、适配流式响应（AI 对话）。
与 WebSocket 区别：SSE 单向、仅文本；WebSocket 双向、支持二进制。
4.5 DNS 域名系统
4.5.1 核心作用
将域名→IP 地址映射，是用户访问网页的第一个协议。
4.5.2 DNS 服务器层级
根 DNS 服务器：13 组 IP，对应全球 1700 + 台服务器，提供顶级域服务器 IP。
顶级域（TLD）服务器：com、cn、org 等后缀服务器，提供权威服务器 IP。
权威 DNS 服务器：存储域名与 IP 的正式映射。
本地 DNS 服务器：运营商提供，代理客户端查询。
4.5.3 DNS 解析流程
浏览器缓存→系统缓存→路由器缓存→本地 DNS 服务器→根服务器→TLD 服务器→权威服务器→返回 IP。
4.5.4 DNS 劫持
定义：篡改 DNS 解析结果，将域名指向恶意 IP。
应对：使用加密 DNS（DoH/DoT）、修改可靠 DNS 服务器、定期检查解析结果。
4.6 输入 URL 到页面展示完整流程
浏览器解析 URL，提取域名。
发起 DNS 解析，获取目标服务器 IP。
浏览器发起 TCP 三次握手，建立连接。
发送 HTTP/HTTPS 请求，携带请求头、Cookie 等。
服务器处理请求，查询资源，生成 HTML 响应。
浏览器接收响应，解析 HTML，渲染 DOM/CSS/JS，加载图片、视频等资源。
页面完全展示后，TCP 四次挥手断开连接（长连接则保持）。
五、补充高频面试考点
TCP 三次握手 / 四次挥手细节：报文标志位（SYN/ACK/FIN）、状态转换、超时处理、2MSL 等待原因。
QUIC 协议核心优势：0-RTT、连接迁移、流间无阻塞、内置加密。
ARP 工作流程：广播请求、单播响应、ARP 缓存。
NAT 类型：静态 NAT、动态 NAT、端口映射 NAT（PAT）。
网络故障排查：PING 测连通性、tracert 测路由、nslookup 测 DNS、arp 查 MAC等。
